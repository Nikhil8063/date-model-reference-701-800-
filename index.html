SELFDEFINING

The SELFDEFINING data type is a concatenated string of tagged data items, where each data item consists of a data type ID followed by its corresponding data.

For SNA topology manager and MultiSystem Manager, SELFDEFINING strings are predefined. Not only does this make the task of building SELFDEFINING values simpler, but they also match the specific needs of the features. These predefined strings are named SELFDEFINING (x) where x is a number (1 - 15).

An important part of these definitions, which follow as separate topics, is the ownership flag or owner identifier. This 3-character value, FLB, is the identifier used by the SNA topology manager. Do notuse FLB when you build SELFDEFINING data types, unless you want to relinquish ownership of the field to the SNA topology manager.

Note:GMFHS-specific fields do not observe the SNA topology manager 3-character ownership convention.

Another important point is that the value in the SELFDEFINING length field is only the length of the SELFDEFINING data. It does not include the length of the SELFDEFINING length field.

example:
8+(N(m+4))

data:

Appendix C. SNA Topology Manager Node Type Transformation

This appendix describes the transformations the SNA topology manager can perform on node objects in the RODM data cache as shown in Figure 22 on page 784. Transformations occur when the SNA topology manager receives topology updates for nodes while it is actively monitoring network or local topology containing the nodes.

The table contains the following headings:

Update Node Type
This column indicates the node type to which the existing node type is to be transformed, and the source of the topology update.
v The following are the node type descriptions:
2.1 Type 2.1 node (object of t2-1Node class).

SN A generic SNA node (object of snaNode class). The SNA topology manager uses this node type only as a place holder, it represents a node that is not being monitored and whose exact node type is not known, but is known to own one or more LUs being monitored. If an update for this node is received, it will be transformed to match the node type reported in the update. The SNA topology manager never transforms an exactly known node type to an snaNode.

LEN A low-entry networking node (object of lenNode class).

EN An end node (object of appnEN class).

BrNN A branch network node (object of appnBrNN class).

MDH This is an end node in the role of migration data host.

NN A network node (object of appnNN class).

BN A network node that has border node capabilities, using its DisplayResourceOtherData field to indicate these capabilities.

ICN A network node in the role of an interchange node.

ICBN An interchange node with border node capabilities indicated in its DisplayResourceOtherData field.

VRN Virtual routing node (connection network).

T4 A (T4) node.

T5 A (T5) node.

v The source of the topology update information is represented by the following:

(v1) Local topology information contained in the vertex-one node data, which pertains to the local (agent) node

(v2)Local topology information contained in the vertex-two node data, which pertains to a node adjacent to the agent node

(net)Network topology information

Note: Refer to IBM SystemView Managed Resource Model Reference and Templates Volume 2: Advanced Peer-to-Peer Networking and SNA Subarea Managementfor information on the meaning of vertex-one and vertex-two nodes in the context of how agents report topology information.

Existing Node Type


This heading represents the set of existing node types subject to change by the topology update. Each column entry contains one of the following action codes:

C Modify the DisplayResourceOtherData (DROD) field to indicate border node capabilities.

ICN Transformed to an interchange node

MDH Transformed to a migration data host

T Complete transformation of the node

— A dash indicates the transformation is ignored.

v Indicates a null matrix intersect where there is no transformation (for example, NN to NN).
